# Phase 2: Execution via AWS Transform Custom

## Purpose

This guide explains how to review and execute the Phase 2 transformation definition generated by Phase 1. The TD is produced automatically by Phase 1 — the user reviews, edits if needed, and runs it.

## Overview

```
Phase 1 Output (Generated TD)
        │
        ▼
┌─────────────────────────────────┐
│  User reviews and edits TD      │  ← Human oversight checkpoint
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│  Execute via AWS Transform      │  ← Local or remote mode
│  custom (with CAST Imaging MCP) │
└─────────────────────────────────┘
```

## Key Difference from Phase 1

Phase 2 **requires CAST Imaging MCP access** during execution. The generated TD instructs the execution agent to:
- Call `object_details` focus="inward" before changing method signatures (check callers)
- Call `transactions_using_object` on every modified object (quantify blast radius)
- Call `quality_insight_violations` after fixes (verify violations addressed)
- Call `architectural_graph` to prevent cross-layer coupling during refactoring
- Call `packages` and `package_interactions` before dependency upgrades

This is NOT a self-contained plan — it's an MCP-augmented transformation.

## Phase 2 TD Structure

The Phase 1 output is already a fully-structured TD with these sections:

1. **Name and Description** — Pattern name and one-line summary
2. **Entry Criteria** — CAST Imaging MCP availability, required runtimes, source code access
3. **Implementation Steps** — The actual transformation work with specific files, objects, and data
4. **CAST Imaging MCP Usage During Execution** — Generic guidance block for the execution agent
5. **Pattern-Specific MCP Verification** — Verification checkpoints specific to the chosen pattern
6. **Validation / Exit Criteria** — Build/test commands, baseline vs target metrics, MCP verification checks

## Execution Modes

### Local Execution (1-9 repositories)

**Prerequisites:**
- AWS CLI configured with `AWSTransformCustomFullAccess` IAM policy
- ATX CLI installed (`curl -fsSL https://transform-cli.awsstatic.com/install.sh | bash`)
- Node.js 22+ and Git installed
- Target language runtime installed locally
- WSL required on Windows
- CAST Imaging MCP server config file available

**Command (with CAST Imaging MCP):**
```bash
atx custom def exec \
  -n "<td-name>" \
  -p "<repo-path>" \
  -x -t \
  --mcp-config ./mcp.json
```

**Options:**
- `-n`: Transformation definition name
- `-p`: Path to the repository
- `-x`: Execute
- `-t`: Include test generation
- `--mcp-config`: Path to MCP server configuration (gives the execution agent live CAST Imaging access)

**MCP Configuration File (`mcp.json`):**
```json
{
  "mcpServers": {
    "cast-imaging": {
      "type": "http",
      "url": "<cast_imaging_mcp_url>",
      "headers": {
        "x-api-key": "${input:imaging-key}"
      }
    }
  }
}
```

**Why `--mcp-config` matters:** Without it, the TD instructions tell the agent *what* to verify but it can't actually call the tools. With it, the agent has live access to `object_details`, `transactions_using_object`, `quality_insight_violations`, etc. during transformation — enabling real-time caller verification, blast radius analysis, and violation resolution tracking.

**Command (without MCP — fallback):**
```bash
atx custom def exec \
  -n "<td-name>" \
  -p "<repo-path>" \
  -x -t
```

This still works — the TD contains enough static context (file paths, object IDs, fix strategies) for the agent to execute. But verification against live CAST Imaging data won't be possible.

### Remote Execution (10+ repositories)

**Prerequisites:**
- Repositories accessible from AWS (CodeCommit, S3, or public Git)
- IAM role with `AWSTransformCustomExecuteTransformations` policy
- MCP config accessible to the execution environment

**Command:**
```bash
atx custom def exec \
  -n "<td-name>" \
  -p "<repo-url>" \
  -x -t \
  --mcp-config ./mcp.json
```

For batch execution across multiple repos, use the AWS Transform web application which provides centralised monitoring and campaign management.

## Passing CAST Imaging Context

The generated TD includes embedded CAST Imaging MCP instructions. During execution, the agent uses MCP tools for:

**Before modifying any file:**
- Retrieve current implementation via `object_details` focus="code"
- Check all callers via `object_details` focus="inward"
- Understand dependencies via `object_details` focus="outward"
- Understand component/layer placement via `architectural_graph`

**During transformation:**
- Quantify blast radius via `transactions_using_object` on EVERY modified object
- Check data entity interactions via `data_graphs_involving_object`
- Trace execution paths via `pathfinder_hierarchy_details`

**After each step:**
- Verify no callers missed via `object_details` focus="inward"
- Confirm flow integrity via `transaction_details` focus="type_graph"
- Cross-reference violations addressed via `quality_insight_violations`

**CAST Imaging MCP must be configured and accessible throughout Phase 2 execution.**

## Workflow for Executing Phase 2

1. **Review the generated TD** — Edit as needed:
   - Add/remove files from scope
   - Adjust target versions
   - Change remediation strategies
   - Add constraints or exclusions

2. **Verify prerequisites:**
   - CAST Imaging MCP server is configured and responding
   - Target language/runtime is installed
   - Repository source code is accessible

3. **Choose execution mode:**
   - 1-9 repos → Local execution
   - 10+ repos → Remote execution via AWS Batch

4. **Run the transformation** — Execute the TD against the repository

5. **Validate results:**
   - Build the project (command from TD's exit criteria)
   - Run tests (command from TD's exit criteria)
   - Review changes (git diff)
   - Run pattern-specific MCP verification checks from the TD
   - Compare quality metrics against baseline from assessment

## Tips

- **Start with a single repo** — Validate the transformation works before scaling
- **Use `--dry-run` first** — See what would change without making changes
- **Keep the TD focused** — A smaller, well-scoped TD executes better than a massive one
- **Phase if needed** — For combined patterns, run each phase as a separate TD execution
- **Version control the TD** — Commit the generated TD alongside the code for traceability
- **CAST Imaging MCP is essential** — The execution agent actively uses it for verification; don't disconnect it during Phase 2
